# Отказоустойчивость

Этот документ описывает защиту на уровне кода, когда один из микросервисов временно недоступен.
Основная точка защиты находится в `gateway`, потому что именно он вызывает остальные сервисы.

## Что реализовано

### Таймауты

Все HTTP-вызовы из `gateway` к зависимым сервисам используют общий клиент с таймаутом `5s`.
Если сервис не отвечает, gateway не зависает бесконечно, а возвращает ошибку клиенту.

Код: `src/gateway/models/main.go`, `src/gateway/models/resilience.go`.

### Retry для безопасных запросов

Повторяются только безопасные операции чтения:

- `GET /flights`;
- `GET /flights/{flightNumber}`;
- `GET /tickets`;
- `GET /tickets/{ticketUid}`;
- `GET /privilege`;
- `GET /requests`.

Retry выполняется до 3 попыток с небольшой паузой между попытками.
Повторяются сетевые ошибки и временные HTTP-статусы `429`, `500`, `502`, `503`, `504`.

Операции изменения состояния не повторяются автоматически:

- создание билета;
- резервирование места;
- освобождение места;
- списание или начисление бонусов;
- удаление билета;
- регистрация пользователя.

Это сделано специально: простой retry для `POST` или `DELETE` может создать дубль билета или дважды изменить баланс.
Для безопасного retry таких операций нужен idempotency key.

### Circuit breaker

Для каждого downstream-сервиса gateway ведет отдельный circuit breaker:

- `flights-service`;
- `tickets-service`;
- `privileges-service`;
- `statistics-service`.

После 3 подряд неудачных обращений circuit открывается на 15 секунд.
Пока circuit открыт, gateway сразу отвечает `503 Service Unavailable` и не нагружает заведомо проблемный сервис.
После паузы gateway снова пробует выполнить запрос.

### Корректные ответы клиенту

Если зависимый сервис недоступен, gateway возвращает:

```http
503 Service Unavailable
```

Вместо прежнего поведения с `panic` или общим `500 Internal Server Error`.

Бизнес-ошибки остаются бизнес-ошибками:

- нет мест на рейс: `409 Conflict`;
- рейс не найден: `404 Not Found`;
- нет или протух JWT: `401 Unauthorized`.

### Компенсация покупки билета

Покупка билета состоит из нескольких шагов:

1. `flights-service` резервирует место.
2. `tickets-service` создает билет.
3. `privileges-service` обновляет бонусную историю.

Если шаг 2 падает после резервирования места, gateway пытается освободить место обратно.
Если шаг 3 падает после создания билета, gateway пытается удалить билет и освободить место.

Это базовая Saga-компенсация для синхронного сценария покупки.

### Статистика и Kafka

Отправка статистики в Kafka выполняется асинхронно после обработки HTTP-запроса.
Если Kafka producer не создан или отправка сообщения падает, gateway логирует ошибку и не ломает пользовательский запрос.
Статистика в таком случае может быть потеряна, но основная операция продолжает выполняться.

## Поведение при падении сервисов

| Недоступен сервис | Что произойдет |
| --- | --- |
| `flights-service` | Нельзя получить список рейсов, найти рейс или купить билет. Gateway вернет `503`. |
| `tickets-service` | Нельзя получить личный кабинет, список билетов, создать или отменить билет. Gateway вернет `503`. |
| `privileges-service` | Личный кабинет и покупка с бонусами вернут `503`; если билет уже создан, gateway попробует откатить билет и место. |
| `statistics-service` | Страница статистики вернет `503`; покупка билета от этого сервиса не зависит. |
| `identity-provider-service` | Вход и регистрация вернут `503`; уже выданные JWT продолжают проверяться локально в gateway. |

## Что остается вне гарантий

Текущая защита лучше прежней, но это еще не полноценная production-схема.

Остающиеся ограничения:

- нет persistent outbox для гарантированного завершения компенсаций;
- если rollback-запрос тоже упал, gateway только логирует ошибку;
- нет idempotency key для безопасного повтора покупки;
- сервисы в Kubernetes запущены в `replicas: 1`, поэтому нет HA на уровне экземпляров;
- базы данных и Kafka не покрыты отдельной логикой деградации внутри сервисов;
- статистика при недоступности Kafka может быть потеряна, потому что нет outbox для повторной доставки.

Для production обычно добавляют:

- idempotency key для `POST /tickets`;
- outbox/inbox pattern для событий и компенсаций;
- persistent Saga state;
- отдельные health/readiness endpoints;
- несколько реплик stateless-сервисов;
- алерты по частым `503` и открытым circuit breaker.

## Проверка вручную

Остановить один сервис:

```bash
kubectl -n rsoi scale deployment/flights --replicas=0
```

Вызвать gateway:

```bash
curl -i 'http://localhost:8080/api/v1/flights?page=1&size=20'
```

Ожидаемый результат: `503 Service Unavailable`.

Вернуть сервис:

```bash
kubectl -n rsoi scale deployment/flights --replicas=1
kubectl -n rsoi rollout status deployment/flights
```

Запустить тесты gateway:

```bash
cd src/gateway
go test ./...
```
